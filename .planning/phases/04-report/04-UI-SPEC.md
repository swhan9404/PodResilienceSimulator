# Phase 4 -- UI Design Contract

> Visual and interaction contract for the Report phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (native HTML + Tailwind utility classes) |
| Icon library | none (text labels and Unicode markers) |
| Font | system-ui, 'Segoe UI', Roboto, sans-serif (inherited from index.css) |

**Source:** Consistent with Phases 2 and 3. No component library per CLAUDE.md "Simplicity First" directive. The report is static HTML/CSS rendered via Tailwind -- no Canvas, no charting library needed.

---

## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gap between timeline dot and line, inline icon gaps |
| sm | 8px | Padding inside timeline event detail, gap between stat card label and value |
| md | 16px | Gap between summary cards, padding inside cards, table cell padding |
| lg | 24px | Vertical gap between report sections (timeline / cards / table) |
| xl | 32px | Report container outer padding (matches right panel `p-8`) |
| 2xl | 48px | Not used in this phase |
| 3xl | 64px | Not used in this phase |

Exceptions: Timeline left padding is 32px (xl) to accommodate the vertical line and dot markers.

**Tailwind mapping:** `gap-1` (4px), `gap-2` (8px), `gap-4` (16px), `gap-6` (24px), `p-8` (32px).

## Typography

| Role | Size | Weight | Line Height | Tailwind Class |
|------|------|--------|-------------|----------------|
| Body / Table cell | 14px | 400 (regular) | 1.5 | `text-sm font-normal` |
| Label / Caption | 12px | 600 (semibold) | 1.5 | `text-xs font-semibold` |

**Weights used: 2 (400 regular, 600 semibold).** The stat value display at 28px achieves sufficient visual differentiation through size alone -- no additional weight distinction needed beyond semibold.

### Timeline Event Marker Colors

| Event | Color | Hex | Usage |
|-------|-------|-----|-------|
| Simulation Start | Blue | `#3B82F6` | First timeline marker (neutral/informational) |
| First Readiness Failure | Amber | `#F59E0B` | Warning -- degradation begins |
| First Liveness Restart | Red | `#EF4444` | Danger -- pods restarting |

**Rationale:** Reuses the existing semantic color palette (podReady green, podNotReady amber, podRestarting red, accent blue) from `src/visualization/colors.ts`. The degradation story is told through color progression: blue (start) -> amber (warning) -> red (failure) -> blue (intervention) -> green (recovery).

## Visual Hierarchy

**Primary focal point:** SummaryCards stat values (28px semibold). These are the "big numbers" the user came to see -- recovery time, error rate, total requests.

**Secondary focal point:** DegradationTimeline event labels (14px semibold) with colored markers telling the degradation story chronologically.

**Supporting information:** ProfileTable (14px regular) and card subtitles (12px regular) provide additional detail on demand.

## Component Inventory

### Component: SimulationReport (container)

The top-level report component, conditionally rendered in App.tsx when `playback === 'recovered'`.

| Property | Value |
|----------|-------|
| Container | `flex-1 p-8 flex flex-col gap-6 overflow-auto` (replaces PodCanvas + MetricsCharts) |
| Heading | "Simulation Report" -- 16px semibold, text-primary |
| Sections (top to bottom) | DegradationTimeline, SummaryCards, ProfileTable |
| Data source | Frozen `ReportData` object from Zustand store (computed once at recovery) |

### Component: DegradationTimeline (RPT-01, RPT-02, RPT-03)

A vertical timeline showing the degradation story from simulation start to recovery.

| Property | Value |
|----------|-------|
| Layout | Vertical list with left-aligned vertical line |
| Container padding-left | 32px (to accommodate line + markers) |
| Vertical line | 2px wide, `var(--border-color)`, positioned `left: 11px` (center of 24px marker area) |
| Line extent | Top of first marker to bottom of last marker |

**Timeline Event Item:**

| Property | Value |
|----------|-------|
| Marker | 12px diameter circle, filled with event color (see color table above) |
| Marker position | Absolute, centered on the vertical line (`left: 6px`, overlapping the line) |
| Event label | 14px semibold, text-primary. Examples: "First Readiness Failure", "All Pods Down" |
| Timestamp | 14px regular, text-secondary, placed right of label with 8px gap. Format: `+{mm:ss}` relative to simulation start |
| Detail text | 12px regular, text-secondary. Additional context line below the label. Example: "Pod 2 removed from load balancer" |
| Item spacing | 24px (lg) between consecutive timeline events |

**Timeline Events (in order):**

| # | Label | Detail | Color | Condition |
|---|-------|--------|-------|-----------|
| 1 | Simulation Started | "Traffic at {rps} RPS with {profileCount} profiles" | Blue `#3B82F6` | Always shown (timestamp = +00:00) |
| 2 | First Readiness Failure | "Pod {id} removed from load balancer" | Amber `#F59E0B` | Shown if `criticalEvents.firstReadinessFailure !== null` |
| 3 | First Pod Restart | "Pod {id} restarted by liveness probe" | Red `#EF4444` | Shown if `criticalEvents.firstLivenessRestart !== null` |
| 4 | Total Service Down | "All pods unavailable -- 100% 503 responses" | Red `#EF4444` | Shown if `criticalEvents.allPodsDown !== null` |
| 5 | Requests Stopped | "Traffic stopped -- recovery in progress" | Blue `#3B82F6` | Shown if `criticalEvents.stopRequestsTime !== null` |
| 6 | Full Recovery | "All pods READY -- service restored" | Green `#22C55E` | Always shown (this is the recovery event) |

**Null events:** If a critical event did not occur during simulation (e.g., pods never fully went down), that timeline entry is omitted entirely. The vertical line connects only existing events.

### Component: SummaryCards (RPT-04, RPT-06)

A grid of large-number cards showing key metrics.

| Property | Value |
|----------|-------|
| Layout | 3-column grid, 16px gap between cards |
| Tailwind | `grid grid-cols-3 gap-4` |
| Card background | `var(--bg-secondary)` |
| Card border | 1px `var(--border-color)` |
| Card border-top | 3px solid accent color (per card) |
| Card border-radius | 8px |
| Card padding | 16px |
| Card alignment | Center-aligned text |

**Card Details:**

| Card | Label | Value Format | Subtitle | Accent Color |
|------|-------|-------------|----------|-------------|
| Recovery Time | "Recovery Time" | `{m}m {s}s` or `{s}s` (if < 60s) | "from stop to full ready" | Green `#22C55E` |

| Total Requests | "Total Requests" | `{count}` (with comma separator via Intl.NumberFormat) | "{dropped} dropped by restart" | Blue `#3B82F6` |

**Card Text Hierarchy:**

| Element | Size | Weight | Color |
|---------|------|--------|-------|
| Label (top) | 12px | 600 (semibold) | `var(--text-secondary)` |
| Value (center) | 28px | 600 (semibold) | `var(--text-primary)` |
| Subtitle (bottom) | 12px | 400 (regular) | `var(--text-secondary)` |
| Vertical spacing | 4px between label/value, 4px between value/subtitle |

### Component: ProfileTable (RPT-05)

A table showing per-profile average response times.

| Property | Value |
|----------|-------|
| Layout | Full-width table below summary cards |
| Table heading | "Response Time by Profile" -- 16px semibold, text-primary, 24px top margin |
| Table background | `var(--bg-secondary)` |
| Table border | 1px `var(--border-color)`, 8px border-radius |
| Header row bg | `var(--bg-secondary)` (same as body in dark mode; slightly differentiated by text weight) |
| Cell padding | 16px horizontal, 8px vertical (`px-4 py-2`) |
| Text alignment | Name column left-aligned, numeric columns right-aligned |

**Table Columns:**

| Column | Header | Width | Align | Content |
|--------|--------|-------|-------|---------|
| Color | (no header) | 16px | Center | 12px filled circle in profile color |
| Profile | "Profile" | auto (fill) | Left | Profile name, 14px regular |
| Avg Response | "Avg Response" | 120px | Right | `{ms}ms` (integer), 14px regular |
| Requests | "Requests" | 100px | Right | `{count}` (with comma separator), 14px regular |

**Row styling:**
- Rows separated by 1px `var(--border-color)` bottom border
- Last row has no bottom border
- No hover effect (static data, not interactive)
- Sort order: by request count descending (highest traffic profile first)

## Interaction Contract

### State Machine Extension

```
stopped_requests --[all pods recovered]--> recovered   (auto, D-01)
recovered --[Reset]--> idle                             (user action, D-03)
```

Added to the existing Phase 3 state machine. No new user-initiated transitions other than Reset.

### Report Lifecycle

1. **Trigger:** Engine `phase` transitions to `'recovered'` (detected in store callback)
2. **Data freeze:** `ReportData` computed once from final snapshot + critical events, stored in Zustand
3. **Render:** App.tsx detects `playback === 'recovered'` and renders `<SimulationReport>`
4. **Dismiss:** User clicks Reset -- `playback` returns to `idle`, `reportData` cleared from store, visualization area restored

### No User Interaction Within Report

The report is entirely read-only. No:
- Clickable elements within the report body
- Expand/collapse
- Sorting controls on the table
- Export buttons
- Scroll-to-event links

The only actionable element is the Reset button in the left panel (existing component).

| Value | Format | Example |
|-------|--------|---------|
